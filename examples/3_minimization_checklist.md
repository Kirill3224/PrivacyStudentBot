# Приклад: Заповнений Чек-ліст для @PKML_policy_bot

**Версія:** 1.0
**Дата:** 07.11.2025

**Філософія:** Це наш "case study". Він показує, як ми застосували наш власний `artifacts/3_minimization_checklist.md` до нашого бота.

---

### Категорія 1: Контроль Доступу (Must-Have)

- [x] **2FA (Двофакторна Автентифікація):** **ВИКОНАНО.** Доступ до хостингу та репозиторію (GitHub) захищений 2FA.
- [x] **Принцип "Найменших привілеїв":** **ВИКОНАНО.** Лише 2 особи (Team Lead, Tech Lead) мають доступ до "секретів" (токена).
- [x] **БЕЗ ПУБЛІЧНИХ ПОСИЛАНЬ:** **ВИКОНАНО (Н/З).** (Не Застосовується). Виконано архітектурно. Ми не використовуємо Google Sheets, тому ризик відсутній.

### Категорія 2: Права Користувачів (Вимоги Закону)

- [x] **Публічна Політика:** **ВИКОНАНО.** Реалізована команда `/privacy`, яка показує нашу політику (див. `2_privacy_policy_example.md`).
- [x] **Механізм Видалення (Ст. 8 Закону):** **ВИКОНАНО (Н/З).** Виконано архітектурно. Наш бот "stateless" (згідно з `1_dpie_lite_example.xlsx`), він не зберігає дані. **Нам немає чого видаляти.** Це ідеальне виконання "права на забуття".
- [x] **Контакт для скарг:** **ВИКОНАНО.** Команда `/help` (або контакти в `/privacy`) веде на @rntroo та @QFITP.

### Категорія 3: Технічна Гігієна (Best Practice)

- [x] **Безпека Токенів:** **ВИКОНАНО (Критично).** Як і вирішено в `1_dpie_lite_example.xlsx`, токен бота зберігається **лише** у змінних оточення (.env) і **не** знаходиться в коді на GitHub.
- [x] **Планування Строків (Retention):** **ВИКОНАНО (Н/З).** Немає даних — немає строків зберігання.
- [x] **Шифрування (Якщо збираєте паролі):** **ВИКОНАНО (Н/З).** Ми не збираємо паролі.
